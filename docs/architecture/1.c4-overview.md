# C4 Model – Architecture Overview

## What is the C4 Model?

The C4 model is a lightweight and flexible framework for describing software architecture. It is suitable for a wide range of architectural styles, including monoliths, microservices, event-driven systems
and domain-driven designs.

The model is based on two core ideas:
- A simple set of abstractions to describe software systems;
- A small set of diagram types that present these abstractions at different levels of detail.

The guiding principle of the C4 model is: 

> Overview first, then zoom and filter, and finally provide details on demand.

Rather than relying on a single complex diagram, the C4 model promotes multiple focused views of the system, each designed for a specific audience and purpose.

---

## Core Abstractions

The C4 model uses a small and consistent vocabulary to describe architecture.

### Person
A person represents an end user or role that interacts with the system.

Examples:
- User;
- Administrator;
- Developer.

### System
A system is the highest-level abstraction that delivers value to users. It is typically owned and maintained by a development team.

Examples:
- ft_transcendence platform (internal system);
- OAuth provider, email service or observability stack (external/internal systems).

### Container
A container represents an independently runnable or deployable unit within a system. This can be an application, service or data store.

> Note: A C4 container is a logical concept and is not necessarily a Docker container, although in this project the two often align.

Examples:
- Frontend web application;
- Backend API service;
- Database;
- Reverse proxy / API Gateway with WAF (ModSecurity);
- Secret management (Vault), Metrics (Prometheus) and Dashboards (Grafana).

### Component
A component is a building block inside a container. It represents cohesive modules, controllers or services that implement specific responsibilities.

Examples:
- Authentication module;
- User management service.
- Matchmaking logic;
- Database access layer.

## C4 Diagram Levels

The C4 model defines four hierarchical diagram levels, each with a specific scope and audience.

### Level 1 — Context Diagram

**Purpose:**
Provides a high-level view of the system, its users and external dependencies.

**Audience** 
- Technical and non-technical stakeholders;
- Evaluators;
- New contributors.

**Scope**
- People (users and roles);
- The main system;
- External systems;
- High-level trust boundaries.

---

### Level 2 — Container Diagram

**Purpose:**
Shows the internal structure of the system by breaking it down into deployable units and data stores and how they communicate.

**Audience**
- Developers;
- Architects;
- DevOps-oriented reviewers.

**Scope**
- Containers (applications and databases);
- Inter-container communication;
- Network and security boundaries.

---

### Level 3 — Component Diagram

**Purpose:**
Zooms into a specific container to explain how it is structured internally.

**Audience**
- Developers;
- Architects.

**Scope**
- Components within a container;
- Responsibilities and interactions.

In this project, component diagrams are created selectively for security-critical containers (e.g. authentication).

---

### Level 4 — Code Diagram

**Purpose:**
Describes implementation-level details, often using UML class diagrams.

**Audience**
- Developers.

**Scope**
- Classes, interfaces and relationships.

This level is intentionally minimized and may be generated from code when needed.

## Why the C4 Model is used in ft_transcendence

The ft_transcendence project involves:
- A multi-service, containerized architecture;
- Clear separation of concerns;
- Security-sensitive components (authentication, authorization, networking, WAF);
- Observability and health monitoring (Prometheus and Grafana);
- Standardized development environment via Vagrant/Ansible;
- Team-based development.

The C4 model is used to:
- Clearly communicate architectural decisions;
- Reduce ambiguity before implementation;
- Align documentation with containerized deployment;
- Support security-by-design principles;
- Provide evaluators with a clear mental model of the system.

---

## How C4 Diagrams are used in this project

- Architecture is designed **before** implementation;
- Diagrams evolve alongside the codebase;
- Each diagram corresponds to real deployable or logical elements;
- Security boundaries are explicitly documented;
- Diagrams are stored as version-controlled documentation in `docs/architecture`.
